## What Went Well

Documentation matched the actual v1 product scope instead of drifting into future features. The README and docs consistently describe `leetcode stats <username>`, supported help commands, public-data dependency, unofficial status, and no credential/session/config storage.

The cross-platform CI story stayed tightly scoped. The workflow validates Windows, macOS, and Linux and runs both `go test ./...` and `go build ./...` without adding release automation too early.

The release guidance is honest about the current state. It documents Homebrew intent for macOS, checksummed GitHub Releases binaries for Windows and Linux, and GoReleaser as the intended future packaging path without adding `.goreleaser.yaml` before packaging work is intentionally scoped.

Automated tests were extended to protect documentation and workflow behavior. This turns high-risk wording drift into a testable surface.

## Challenges and Friction

The biggest process gap is artifact tracking rather than code correctness. `sprint-status.yaml` is missing, and story files still say `Status: review` even though the task lists, completion notes, tests, build, and manual output check all support readiness.

The project is also operating without Git metadata in this workspace. That limits traceability for commit-based review, rollback, and release provenance.

Documentation tests caught wording gaps during implementation, including a missing literal `Homebrew` wording in README. That is a healthy catch, but it shows that release/trust language is exact enough to deserve continued automated coverage.

## Key Insights

1. Scope discipline was the main success pattern. Epic 3 repeatedly avoided adding auth, config, JSON, dashboard, and release automation before they were scoped.
2. Documentation is now part of the product contract, not a passive appendix. Tests enforce command examples, trust boundaries, and installation expectations.
3. Manual output verification from Adi complements automated verification. The code and output are considered working based on both local checks and project-lead review.
4. Process metadata needs cleanup. Missing sprint status and review-labeled stories can create false uncertainty after the implementation itself is working.
